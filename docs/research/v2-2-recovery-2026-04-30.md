## Q9 — MCP Failure Mode Taxonomy

### Framework 1: Playwright MCP (microsoft/playwright-mcp)

**Failure mode A — Browser profile lock (F-V8 root cause)**

Exact error from `anthropics/claude-code#24144`:
> "browserType.launchPersistentContext: Failed to launch the browser process. [pid=25256][out] Opening in existing browser session. [pid=25256] <process did exit: exitCode=0, signal=null>"

Root cause: Chrome on Windows "does not support multiple simultaneous instances with different user data directories. When Playwright attempts to launch Chrome with its own `--user-data-dir` and `--remote-debugging-port`, Chrome detects the existing instance, attaches to it, and immediately exits." (verified 2026-05-01, `anthropics/claude-code#24144`)

Official status: **Issue closed as "not planned"** — no official fix. Only workaround documented: "Close all Chrome windows before using Playwright MCP tools, then reopen Chrome afterward."

**Failure mode B — Corrupted install lock (microsoft/playwright-mcp#1305)**

Exact error sequence verified 2026-05-01:
1. `"browserType.launchPersistentContext: Chromium distribution 'chrome' is not found at /opt/google/chrome/chrome"`
2. After install: `"No open tabs. Use the 'browser_navigate' tool to navigate to a page first."`
3. Stuck state: `"Browser is already in use for /home/vscode/.cache/ms-playwright/mcp-chrome-76c1264, use --isolated to run multiple instances"`

Root cause: "`browser_install` tool launching a browser instance during installation that enters a corrupted state — holding a profile directory lock while remaining unusable." Maintainer note: "The only fix appears to be restarting the MCP server." (verified 2026-05-01, `microsoft/playwright-mcp#1305`)

**Failure mode C — --isolated flag namespace error in containers**

The `--isolated` flag workaround "fails with namespace errors in containerized environments, leaving the user without a viable solution in their specific devcontainer setup." (verified 2026-05-01, `microsoft/playwright-mcp#891`)

**Recovery paths verified:**
- (a) Kill Chrome processes matching `ms-playwright/mcp-chrome` user-data-dir, delete lockfile — fails if lockfile is "Device or resource busy" (F-V8 empirical evidence)
- (b) `--isolated` flag — works outside containers; fails in devcontainer/namespace-restricted environments
- (c) MCP server restart — the only universally reliable recovery (maintainer-confirmed)
- (d) Use `PLAYWRIGHT_MCP_USER_DATA_DIR` env var to point to a clean temp directory on fresh session

### Framework 2: Supabase MCP (supabase-community/supabase-mcp)

**Failure mode A — Authentication / project lock (supabase-community/supabase-mcp#81)**

Error: "Unauthorized" on `mcp_supabase_apply_migration`. Reporter: "i keep in generating new tokens but the issue persists." Root cause: token scope mismatch or stale auth binding — regenerating tokens does not clear the server-side session state. (verified 2026-05-01, `supabase-community/supabase-mcp#81`)

**Failure mode B — Migration conflict (concurrent push)**

From secondary search synthesis (mcpcat.io): "Migration files are applied in timestamp order, so concurrent pushes from different machines can cause conflicts." The Supabase MCP server has had a `lock_timeout` setting since v0.3.8 for migration execution. Rate-limiting middleware was added to address "excessive queries from imprecise prompts."

**Failure mode C — Advisor staleness**

`get_advisors` tool returns cached advisory data. Stale advisors can misguide the database-engineer agent into applying recommendations that are already resolved or inapplicable to the current schema state. (secondary — search synthesis, no primary issue URL; tagged accordingly)

**Recovery paths:**
- Auth failure: disconnect MCP server from Claude Code settings, re-authorize OAuth, reconnect
- Migration lock: add `lock_timeout` parameter to migration SQL, serialize concurrent push operations
- Advisor staleness: call `get_advisors` immediately before acting on results, never act on cached session output

### Framework 3: GitHub MCP (github/github-mcp-server)

**Failure mode A — Token expiry mid-workflow (anthropics/claude-code#5706)**

Exact quote: "Currently there is no way for Claude Code to refresh tokens for MCP servers. This means that we either have to have a very long lived token or manually refresh it." Issue labeled `area:auth`, `area:mcp`. Status: open, no assignee. (verified 2026-05-01, `anthropics/claude-code#5706`)

MCP spec gap confirmed by reporter: "The MCP spec does seem to call for a refresh token ability but I don't think Claude Code implements it."

**Failure mode B — Scope mismatch**

From `github/github-mcp-server#1683`: "Even when invited to a private repository, my GitHub MCP token could not be used to access issues or pull requests within that project." GitHub Apps "don't use OAuth scopes — they use app-level permissions configured in the GitHub UI, and the X-OAuth-Scopes header comes back empty." MCP Specification added scope handling in November 2025 "allowing clients to up-scope tokens interactively when a server returns a challenge for tools requiring higher scopes." (verified 2026-05-01, secondary via WebSearch synthesis of `github/github-mcp-server#1683`)

**Failure mode C — Rate limit (secondary)**

GitHub API: limit of 10 tokens per user/application/scope combination; 10 tokens created per hour. Oldest tokens auto-revoked when limit exceeded. User access tokens from GitHub Apps expire after 8 hours by default and require refresh token exchange. (verified via WebSearch synthesis, `docs.github.com/en/authentication/keeping-your-account-and-data-secure/token-expiration-and-revocation`)

**Recovery paths:**
- Token expiry: pre-emptive refresh before 8-hour window; use long-lived PAT for automation workflows as interim
- Scope mismatch: verify GitHub App permissions UI settings match required scopes; use up-scope flow when MCP spec challenge is implemented
- Rate limit: add jitter to retry, honor `X-RateLimit-Reset` header value

### Framework 4: Context7 MCP (upstash/context7)

**Failure mode A — Rate limiting (upstash/context7#808)**

Error: "too many request" — reporter received this "despite the user's first request in 12 hours." Issue closed without maintainer response. No documented quota thresholds. (verified 2026-05-01, `upstash/context7#808`)

**Failure mode B — Tool resolution failure (upstash/context7#30)**

Error: `MCP error -32602` — "Tool resolve_library_id not found." This is a protocol-level error (invalid params) returned when the server does not expose the tool or the tool name is mismatched between client expectations and server registration. (verified 2026-05-01, `upstash/context7#30`)

**Failure mode C — Connection drop / context deadline (upstash/context7#1508)**

Error: "context deadline exceeded, sse fallback failed." SSE transport timeout with no graceful reconnect. Related: `upstash/context7#1473` — "Connection closed" (-32000) on tool invocation despite server advertising tools. (verified 2026-05-01, secondary via WebSearch synthesis)

**Recovery paths:**
- Rate limit: wait-and-retry with exponential backoff; fall back to direct WebSearch on rate-limit signal
- Tool resolution failure: verify MCP server version matches client expectations; call `list_tools` first to confirm tool names
- Connection drop: reconnect via `/mcp reload` in Claude Code; fall back to WebFetch for doc lookups

### Framework 5: Vercel MCP

**Failure mode A — Rate limit (file upload, API)**

From Vercel docs + secondary synthesis: rate limits are "hard limits that apply to the platform when performing actions that require a response from our API." Hitting file upload rate limit can result in a "24-hour lockout." Large Next.js deployments (>15,000 files) can generate 40,000+ API requests per deploy. (verified 2026-05-01, secondary via WebSearch synthesis of `vercel.com/docs/limits`)

**Failure mode B — Deploy protection blocking preview MCP**

Preview deployments with Deployment Protection require authentication to access preview MCP server endpoints. The MCP client receives a 401 when attempting to call tools on an unauthed preview deployment. Recovery: use `Instant Rollback` to revert, or configure deployment protection to allow MCP access. (secondary — Vercel docs synthesis)

**Recovery paths:**
- Rate limit: implement batched file upload with CLI-level retry (similar to AWS SDK backoff pattern)
- Deploy protection: explicitly configure `deploymentProtection` to allow internal tool access for MCP preview servers

## Q10 — Health Check Strategy Comparison

### Three structural analogues

**Analogue 1: Kubernetes readiness + liveness probes**

From kubernetes.io (verified 2026-05-01, `kubernetes.io/docs/concepts/configuration/liveness-readiness-startup-probes/`):
> "Kubernetes provides readiness probes to detect and mitigate situations where applications are temporarily unable to serve traffic. If the readiness probe fails, the EndpointSlice controller removes the Pod's IP address from the EndpointSlices of all Services that match the Pod."

> "If the liveness probe fails, the kubelet kills the container, and the container is subjected to its restart policy."

K8s uses **three probe types** with distinct semantics:
- **Startup probe** — allows initialization before health gates apply
- **Readiness probe** — controls traffic routing (analogous to "is the MCP server ready to serve tools?")
- **Liveness probe** — triggers restart when the server is stuck (analogous to "is the MCP server's event loop unblocked?")

Key discipline: "A common pattern for liveness probes is to use the same low-cost HTTP endpoint as for readiness probes, but with a higher `failureThreshold`, which ensures that the pod is observed as not-ready for some period of time before it is hard killed." (verified 2026-05-01, secondary synthesis of k8s best-practices)

**Analogue 2: MCP Protocol ping utility**

From `modelcontextprotocol.io/specification/2025-03-26/basic/utilities/ping` (verified 2026-05-01):
> "The receiver MUST respond promptly with an empty response. If no response is received within a reasonable timeout period, the sender MAY: Consider the connection stale / Terminate the connection / Attempt reconnection procedures."

> "Implementations SHOULD periodically issue pings to detect connection health. The frequency of pings SHOULD be configurable. Timeouts SHOULD be appropriate for the network environment."

> "Timeouts SHOULD be treated as connection failures. Multiple failed pings MAY trigger connection reset. Implementations SHOULD log ping failures for diagnostics."

The spec defines ping as a standard JSON-RPC request with `"method": "ping"` and no parameters. A health check guide (fast.io, verified 2026-05-01) recommends "sending `{'jsonrpc': '2.0', 'method': 'ping', 'id': 'health-check-1'}` every 30 seconds with a 2-5 second timeout window." The same guide proposes `list_tools` as a functional probe: "calling `list_tools` on each server — the same handshake your agent uses — so a green status means the server is actually ready to serve MCP requests."

**Analogue 3: MCP error-handling three-tier architecture**

From mcpcat.io error-handling guide (verified 2026-05-01):
> "Return structured error responses using the `isError` flag" — the mechanism "prevents server crashes and enables LLMs to understand failures and retry operations."

Three recovery strategies defined:
> "Only retry idempotent operations (safe to repeat)" with exponential backoff and jitter to prevent synchronized retry storms. / Circuit Breakers: Prevent cascading failures through three states (Closed, Open, Half-Open). / Graceful Degradation: Return cached data or partial functionality when dependencies fail.

Rate limiting specific guidance:
> "Return clear rate limit error messages" and "Include retry-after information when possible."

### Option A: PostSessionStart hook pinging each enabled MCP server

**What it does:** At session start, the hook calls `list_tools` (or MCP `ping`) on each MCP server listed in the project's `CONFIG.yaml` `mcp_plugins` slot. Returns a per-server health status injected into the session context as a system reminder.

**Maintenance cost:**
- Requires `mcp_plugins` slot in `CONFIG.yaml` (FD-02 sub-question — not yet resolved)
- Hook must be versioned separately from skill bodies
- Every new MCP server requires a hook update (N+1 maintenance)
- False positives possible: a server that passes ping may still fail on specific tool calls (e.g., Supabase passes ping but returns 401 on `apply_migration`)
- K8s analogy: **startup probe** — tells you the server process is alive, not that all tools are functional

**Precedent:** `github.com/anthropics/claude-code/issues/29626` — "Add automatic MCP server health check on session start" feature request. No implementation as of 2026-05-01. Commentary: "A SessionStart hook with directive message can be used, though it's unreliable because the AI doesn't always act on hook output." (verified 2026-05-01, secondary synthesis)

**Verdict:** Moderate maintenance cost. Valuable for connection-class failures (server not running, transport down). NOT sufficient for tool-class failures (auth, lock, rate limit). Requires `mcp_plugins` config slot to exist first.

---

### Option B: Per-tool "Common Recovery" prose in each skill body

**What it does:** Each skill that references an MCP tool includes a "Common Recovery" subsection documenting the failure modes and exact recovery steps for that tool. Already the pattern for F-V8 in `browser-driven-verification` (the research item that prompted this workstream).

**Maintenance cost:**
- No infrastructure dependency — docs only
- Recovery steps live where the skill is used (locality of reference)
- Skills already have a "Red Flags" and "Anti-Patterns" pattern for similar content
- Cost to add: ~15-30 lines per skill body per MCP server referenced
- Staleness risk: if the MCP server changes its error messages or behavior, the prose may drift
- K8s analogy: **runbook entry** — not a probe, but a human-readable remediation path per symptom

**Precedent:** F-V8 ACTION in PROJECT-PLAN.md: "add a 'Common Recovery' section to `browser-driven-verification` skill body documenting the lock-fail mode + two recovery paths." This is the established PF pattern for documenting recoverable failures at the skill level.

### Comparison Table

| Axis | PostSessionStart hook ping | Per-tool skill prose |
|---|---|---|
| Failure classes caught | Transport/connection only | All classes (transport + auth + lock + rate limit + tool-specific) |
| Recovery action | Automated alert before tools are used | Human-readable steps at point of failure |
| Maintenance cost | HIGH — requires config slot + hook versioning + per-server update | LOW — docs-only; lives with the skill |
| False positive risk | Moderate (ping passes, tool fails) | None (prose is always accurate to the scenario it documents) |
| Infrastructure dependency | Requires `CONFIG.yaml` `mcp_plugins` slot (FD-02 unresolved) | None |
| K8s analogue | Readiness probe (live/dead signal) | Runbook entry (symptom → remediation) |
| Right answer for | "Is the server running?" | "The server is running but tool X failed — what do I do?" |
| PF precedent | No — feature-request only (CC#29626) | Yes — F-V8 action, existing Anti-Patterns/Red-Flags pattern in all skills |

**Synthesis:** The two options are complementary, not competing. K8s runs BOTH readiness probes AND runbooks. The failure modes documented in Q9 split cleanly: connection-class failures (server not running, transport down, connection drop) are probe-catchable; auth/lock/rate-limit failures are only recoverable via per-tool prose because the server is live and responding. The 12 failure modes in Q9 break down as:
- 2 catchable by ping (Playwright corrupted install, Context7 connection drop)
- 10 NOT catchable by ping (all auth, lock, rate-limit, scope, tool-resolution failures)

**Recommendation for Q10:** Implement per-tool "Common Recovery" prose **first** (low cost, high coverage). Defer PostSessionStart ping hook until FD-02 (`mcp_plugins` config slot) is resolved. The ping hook is a nice-to-have optimization; per-tool prose is the load-bearing recovery surface.

## Q11 — Systematic Discovery via trigger-audit.jsonl

### Existing infrastructure (already shipped)

The PF v2 `pre-tool-use` hook already writes to `.framework-state/trigger-audit.jsonl` at `hooks/pre-tool-use` line 114:

```bash
printf '{"timestamp":"%s","event":"%s","name":"%s"}\n' "$ts" "$event_type" "$name" >> "${STATE_DIR}/trigger-audit.jsonl"
```

Current schema: `{timestamp, event, name}` — narrow payload for Skill and Agent invocations only.

This is a **starting point**, not a complete MCP error audit schema.

### Recommended extended schema for MCP tool errors

Drawing from three structured logging precedents:

**Precedent 1: Mattermost audit log schema** (verified 2026-05-01, `docs.mattermost.com/comply/embedded-json-audit-log-schema.html`)

A standard audit log entry contains: `timestamp, event_name, status, actor information (user_id, session_id, client, IP address), event details with parameters and state information, metadata (API path, cluster ID), and an error object.`

**Precedent 2: MCP error-handling three-tier classification** (verified 2026-05-01, mcpcat.io):

Three error tiers: transport-level, protocol-level (JSON-RPC error codes), application-level (isError flag + tool result). Each tier needs different fields.

**Precedent 3: Microservices audit logging pattern** (microservices.io, verified 2026-05-01):

"Record user activity in a database." Drawback acknowledged: "The auditing code is intertwined with the business logic, which makes the business logic more complicated" — the alternative of Event Sourcing as "a reliable way to implement auditing."

**Recommended JSONL schema extension:**

```jsonl
{
  "timestamp": "ISO-8601 UTC",
  "event": "mcp_tool_error",
  "mcp_server": "playwright | supabase | github | context7 | vercel | notion",
  "tool_name": "browser_navigate | apply_migration | create_issue | ...",
  "error_class": "transport | auth | lock | rate_limit | tool_resolution | unknown",
  "error_message": "verbatim error string from MCP tool result",
  "is_error_flag": true,
  "retry_count": 0,
  "recovered": false,
  "recovery_action": "mcp_restart | token_refresh | backoff_retry | manual | none",
  "session_id": "from session.json session_started_at",
  "skill_context": "browser-driven-verification | rls-aware-migrations | ..."
}
```

**Key fields:**
- `error_class` — enables the Post-Mortem agent to mine by class (e.g., "show me all `lock` class errors this month")
- `recovered` + `recovery_action` — tracks whether the documented recovery prose is actually being followed; enables feedback loop
- `skill_context` — links the error to the skill that was active when it occurred; enables per-skill failure rate analysis
- `mcp_server` + `tool_name` — the primary grouping axes for pattern detection

**PostToolUse hook extension (non-breaking):** The existing `pre-tool-use` hook's `log_invocation` function can be extended to also write errors by adding a PostToolUse handler. This was deferred in D-F (PostToolUse hook deferred to v2.1 expansion). The schema above is designed to be appended to the same `trigger-audit.jsonl` file — the existing narrow `{timestamp, event, name}` entries and the extended error entries coexist in the same log (different `event` values).

**Mining pattern for Post-Mortem agent:**
- Scan `trigger-audit.jsonl` for entries where `event = "mcp_tool_error"`
- Group by `mcp_server + error_class` to find repeat failure clusters
- Check `recovered = false` entries for patterns where the documented recovery prose is not working
- Promote to pattern row in `STACK-PATTERNS.md` when the same `mcp_server + tool_name + error_class` triple appears ≥3 times across sessions

## Synthesis

**Q9 consensus (5/5 MCP servers documented):** Every MCP server has at least one auth-class failure and one rate-limit-class failure. Lock failures are Playwright-specific (process-model). Tool-resolution failures are Context7-specific (server registration). The dominant recovery action across all servers is: *the agent cannot self-recover from auth/scope failures — manual human intervention is required.*

**Q10 consensus (3/3 structural analogues agree):** Per-tool prose and probe-based health checks are complementary, not competing. K8s runs both. The probe tells you the server is alive; the runbook tells you what to do when a specific tool fails. For PF v2, per-tool skill prose has lower maintenance cost and higher failure-class coverage than a PostSessionStart ping hook, and has an existing PF precedent (F-V8).

**Q11 consensus (3/3 logging precedents):** Structured JSONL with error classification is the load-bearing pattern. The field that creates analytical leverage is `error_class` (taxonomy-based grouping), not just verbatim error strings.

### R2 — trigger-audit.jsonl schema extension (Q11)

Extend the existing `pre-tool-use` `log_invocation` function to also record MCP tool errors with the schema defined above. Add a PostToolUse hook handler (part of D-F expansion deferred to v2.1) that captures `is_error_flag = true` tool results. The Post-Mortem agent's prompt should be updated to mine `trigger-audit.jsonl` for `event = "mcp_tool_error"` entries and report repeat failure clusters in its Phase 5 retro output.

### R3 — Playwright-specific: MCP restart as first recovery step

The three independently-confirmed Playwright failure modes (Issues #891, #1305, #24144) all have one common reliable recovery: **restart the MCP server**. The `browser-driven-verification` Common Recovery section should document this as the first-line recovery above the process-kill path, because process-kill only works when the lockfile is not "Device or resource busy."
