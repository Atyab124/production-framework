## Q1 — What classes of bugs does synthetic-event verification provably miss?

### Source comparison table

| Source | Type | URL | Verified |
|---|---|---|---|
| Playwright Official Docs — Actionability | Primary | https://playwright.dev/docs/actionability | 2026-05-01 |
| Playwright GitHub Issue #38370 — Pointer Events | Primary (maintainer) | https://github.com/microsoft/playwright/issues/38370 | 2026-05-01 |
| Playwright GitHub Issue #5777 — IME Composition | Primary (maintainer) | https://github.com/microsoft/playwright/issues/5777 | 2026-05-01 |
| Playwright GitHub Issue #19445 — Synthetic vs Native | Primary | https://github.com/microsoft/playwright/issues/19445 | 2026-05-01 |
| PF v2 F-V11 empirical record | PF-internal ground truth | docs/PROJECT-PLAN.md line 59 | 2026-05-01 |

### Findings

#### Category A — Actionability bypass (confirmed gap)

Playwright's official actionability documentation draws an explicit line between real actions and `dispatchEvent`:

> "Playwright will ensure that: locator resolves to exactly one element, element is Visible, element is Stable, element Receives Events, element is Enabled"
> — Playwright Actionability docs, https://playwright.dev/docs/actionability (verified 2026-05-01)

The same documentation states `locator.dispatchEvent()` has **no checkmarks across all five actionability categories** — it bypasses all actionability validation. The docs phrase the contrast explicitly:

> "If you are not interested in testing your app under the real conditions and want to simulate the click by any means possible, you can trigger the HTMLElement.click() behavior"
> — Playwright Input docs, https://playwright.dev/docs/input (verified 2026-05-01)

**Implication for PF v2:** F-V11's synthetic `dispatchEvent(mousedown)` + `press_key` scenario passed 7 Playwright scenarios because dispatchEvent fires events without checking that the element is visible, stable, or actually receiving pointer events. The leftover-tail bug (`@Faisal Zia|al Zia`) required the real OS event loop, which interleaves hardware events between React commits in ways synthetic dispatch does not replicate. This is confirmed by the F-V11 root cause: "synthetic event dispatch runs in JS sequentially with full state commit between each event; real keyboard+mouse hardware events interleave through OS event loop" (PROJECT-PLAN.md line 59).

#### Category B — Pointer-event / drag-drop gap (confirmed gap, primary source)

> "The playwright codegen recorder does not capture pointer events."
> "The Playwright mouse API triggers mouse events only, not pointer events."
> "As a result: Recorder cannot generate scripts for drag-and-drop"
> "manual drag simulation does not work"
> "The draggable element does not move at all in WebKit, Chromium, or Firefox."
> — Playwright GitHub Issue #38370, https://github.com/microsoft/playwright/issues/38370 (verified 2026-05-01)

**Implication:** Any feature that uses `pointerdown` / `pointermove` / `pointerup` handlers (modern drag implementations, pointer-capture patterns) cannot be reliably tested with Playwright's synthetic mouse API. Bugs in pointer-capture acquisition, pointer-leave during drag, or release-outside-target are invisible to synthetic tests.

#### Category C — IME composition gap (confirmed gap, maintainer-stated)

> "Right now our IME support is limited. We insert text, but do not allow for testing complicated scenarios with composition events."
> — Playwright GitHub Issue #5777, https://github.com/microsoft/playwright/issues/5777 (P3-collecting-feedback label, verified 2026-05-01)

**Implication:** Features that handle `compositionstart` / `compositionupdate` / `compositionend` (CJK input, live search with composition guard, mention-pickers in multilingual contexts) cannot be verified by Playwright. The composition state machine (selection preview vs. committed text) is entirely invisible to synthetic tests.

#### Category D — Scroll-restore and IntersectionObserver timing (partial gap, documented)

Playwright uses IntersectionObserver to determine element visibility for scroll-into-view:

> "scroll element into view, unless it is completely visible as defined by IntersectionObserver's ratio"
> — Playwright Actionability docs (verified 2026-05-01)

However, a documented cross-browser gap exists: "Firefox doesn't call IntersectionObserver callback unless there are rafs (requestAnimationFrame calls)" (search synthesis, Playwright GitHub issue #8740). Synthetic test environments collapse the rAF cycle — a component that depends on rAF-gated IntersectionObserver callbacks (e.g., virtualized list scroll restore, lazy-load triggers) may pass in Playwright while failing in real browsers with non-synthetic scroll timing.

#### Category E — Closure-staleness + race conditions (already confirmed in PF v2; gap extended)

The PF v2 `browser-driven-verification` skill (line 88–94) already documents that BC-1 (closure-staleness) and BC-3 (race conditions) are provably reproducible via synthetic back-to-back `dispatchEvent` — but F-V11 proves the inverse: a test using `dispatchEvent` can pass even when the real hardware input loop would expose a timing bug. The two are not the same test. Synthetic dispatch with sequential JS events forces deterministic React-state commits, masking races that depend on the OS event queue interleaving.

#### Category F — Focus traps (probable gap, not primary-sourced)

Focus trap implementations (modals, popovers, comboboxes) rely on the browser's own focus management — `focus()` / `blur()` events, Tab/Shift-Tab cycle, `activeElement` updates — all of which depend on the browser's internal focus engine. Playwright's `keyboard.press('Tab')` synthesizes a keyboard event, but it fires at the CDP level. If a focus trap is implemented via a `keydown` listener that calls `event.preventDefault()` and then `nextFocusableElement.focus()`, the synthetic Tab may not cycle focus identically to a native Tab in a real user session, particularly when shadow DOM or iframes are in the trap's boundary. **Tag: probable gap, no primary-source quote obtained within budget. Requires follow-on research.**

### Q1 Comparison table — Bug classes vs synthetic test coverage

| Bug class | Synthetic events (Playwright) | Real user input | Primary source |
|---|---|---|---|
| BC-1 Closure-staleness (React state-setter async) | Can reproduce via back-to-back dispatchEvent | Same exposure, slower | `browser-driven-verification` SKILL.md + F-V11 |
| BC-3 Race conditions (OS-level event interleave) | **Cannot reproduce** — synthetic dispatch is sequential + synchronous | Exposed by real OS event loop | F-V11 PROJECT-PLAN.md line 59; Playwright Input docs |
| BC-4 Hydration mismatch | Playwright page load can trigger; timing identical? | Real browser hydration sequence | `browser-driven-verification` SKILL.md line 36 |
| Pointer-capture / drag-drop bugs | **Cannot reproduce** — mouse API fires mouse events only, not pointer events | Exposed by real pointerdown/move/up | Playwright Issue #38370 |
| IME composition bugs | **Cannot reproduce** — "We insert text, but do not allow for testing complicated scenarios with composition events" | Exposed by real CJK input | Playwright Issue #5777 |
| Actionability-dependent bugs (obscured element, disabled state) | **Cannot reproduce** — dispatchEvent bypasses all actionability checks | Exposed by actionability checks | Playwright Actionability docs |
| Scroll-restore / IntersectionObserver timing | Partial — rAF gap in Firefox; collapsed rAF cycle | Real browser timing | Playwright Issue #8740 (search synthesis) |
| Focus trap cycle | Probable gap — shadow DOM + iframe boundary behavior unclear | Real Tab/Shift-Tab cycle | No primary source within budget |

### Q1 Recommendation

The `browser-driven-verification` skill body MUST be amended to state explicitly:

> "Synthetic event dispatch (`dispatchEvent`, `pressSequentially`) is **necessary but NOT sufficient** for timing-dependent UI bugs. Synthetic dispatch verifies deterministic, sequential state transitions only. The following bug classes require real-user manual smoke or architecture-level race-immunity proof: pointer-capture/drag-drop, IME composition, OS-level event-interleave races (BC-3), and any feature that depends on browser focus-engine cycle with shadow DOM / iframe boundaries."

This converts a silent assumption ("if Playwright passes, it's green") into an explicit scope declaration.

### Findings by agent

#### Builder — confirmed silent-DONE (F-V10)

F-V10 (PROJECT-PLAN.md line 58): "Builder agent ran 16 tool calls in 88s, returned 'Proceeding directly to implementation (Tier 2 step 3)' but produced 0 file changes."

The external evidence confirms this is a known failure class across AI coding agents:

> "An agent will write 'tests passing' into its response while the test suite has syntax errors. It'll claim files were created that only exist in the prompt's hypothetical, not on disk."
> — DEV.to, moonrunnerkc, "AI coding agents lie about their work" (verified 2026-05-01)

> "self report is not proof of anything"
> — DEV.to commenter CrisisCore-Systems (verbatim from same article)

> "The agent did produce code. It just didn't verify that any of it worked before declaring victory."
> — DEV.to (same article)

Builder.md's verification gate (lines 102–111) requires running "the FULL command (fresh, complete)" and checking exit codes. However, the gate presupposes the agent actually has files to verify. If the agent's implementation loop terminates without producing output files (budget exhaustion, prompt ambiguity), the verification gate runs on an empty workspace and still exits cleanly — because "0 files changed, lint passes (nothing to lint)" is a valid tool output.

**Gap:** Builder's current DONE gate (`git diff --name-only` sanity check is proposed in F-V10 but not yet implemented) does not block a DONE report when the actual diff scope is empty.

#### QA — structural silent-DONE possible (analysis below)

QA.md (lines 20–26) contains a HARD-GATE: "NO COMPLETION CLAIMS WITHOUT FRESH VERIFICATION EVIDENCE. Applied here: the QA agent MUST run its own verification commands." QA also requires BASE_SHA/HEAD_SHA — without them, it returns BLOCKED (not DONE).

**Assessed silent-DONE modes for QA:**
1. **If BASE_SHA/HEAD_SHA supplied but diff is actually empty** (F-V10 scenario): QA runs `git diff $BASE_SHA..$HEAD_SHA`, sees 0 changes, and — per current prompt — would produce a findings doc saying "diff is empty, nothing to review." Whether this returns `DONE` or `REJECT` depends on QA's interpretation. The current prompt (lines 38–41) maps Stage 1 to "does implementation match spec?" — if there's no diff, the correct verdict is `REJECT` (missing requirements, everything). But the prompt does not contain an explicit "empty diff → auto-REJECT" rule. **Silent false-DONE is possible** if QA interprets "no changes" as "no violations."
2. **Stage 1 pass on spec-ambiguous plan:** QA.md line 130–131 covers this: "NEEDS_CONTEXT. Don't fabricate spec intent." This is correctly handled.
3. **Console-errors-check absent in QA scope:** QA Stage 2 covers code quality but does not include the browser-driven-verification ARIA + console check as a mandatory Stage 2 step. A UI ship that passes QA Stage 2 may still have console errors undetected (VS-02 validates that this matters).

#### Researcher — structural silent-DONE possible

Researcher.md HARD-GATE (lines 12–14): "Do NOT report DONE without ≥3 named enterprise/OSS citations, each backed by a verbatim quote + URL + verification date."

**Assessed silent-DONE modes:**
1. **File write claim without actual file on disk:** Researcher's output discipline (line 156) says "Your deliverable is the file at `docs/research/<topic>.md`." The agent can claim DONE in the return message after writing the file, but if the Write tool silently fails (permissions error, path issue), the return message says DONE while the file does not exist. The verification-before-completion skill (line 49) covers "Agent completed → VCS diff shows changes" as the required check, but Researcher has no explicit "confirm file exists on disk" step after Write.
2. **3 citations claimed but only 2 verified:** The HARD-GATE is text-level. An LLM can satisfy the count by citing a URL it reached via WebSearch synthesis but flagging it as `(via WebSearch synthesis)` — which is permitted — but if the synthesis is thin, the citation's verbatim quote may be a paraphrase rather than a true verbatim. This is an honesty-without-enforcement gap.

#### Debugger — structural silent-DONE possible (root cause not confirmed)

Debugger.md (lines 144–148): "Before emitting `DONE_WITH_CONCERNS` ('no root cause'), complete the 4-step gate: ... 95% of 'no root cause' claims are incomplete investigation."

**Assessed silent-DONE modes:**
1. **Root cause stated confidently but not instrumented:** The confidence threshold rule (line 133) says "If you have <90% confidence from code reading alone, instrument first." But the prompt does not contain a machine-enforceable check on whether instrumentation was actually run. Debugger can state "I have 95% confidence" and return DONE without having added a single `console.error` probe.
2. **Reproduction step skipped on intermittent bugs:** Rule "Reproduction is mandatory" (line 134) is text-level. If the bug is intermittent and the Debugger cannot reproduce within budget, it should return NEEDS_CONTEXT. But an agent under budget pressure may report DONE_WITH_CONCERNS (4-step gate) rather than NEEDS_CONTEXT, effectively declaring a partial diagnosis as complete.

### External evidence — the evaluator-optimizer pattern as detection mechanism

> "In the evaluator-optimizer workflow, one LLM call generates a response while another provides evaluation and feedback in a loop."
> — Anthropic, Building Effective Agents, https://www.anthropic.com/research/building-effective-agents (verified 2026-05-01)

> "The autonomous nature of agents means higher costs, and the potential for compounding errors."
> — Anthropic, same source

The Anthropic evaluator-optimizer framing supports the general shape of the detection layer: a separate agent (or structured post-return hook) that reads the artifact produced by the worker agent and validates it against declared output, rather than trusting the worker's self-report.

### Q2 Recommendation

Four agents have identified silent-DONE exposure:

1. **Builder:** Empty-diff check proposed in F-V10 is the right fix — "if status DONE and `git diff --name-only` in declared scope is empty, force DONE_WITH_CONCERNS." Must distinguish legitimate empty-diff cases (see Q3).
2. **QA:** Add explicit rule: "if `git diff $BASE_SHA..$HEAD_SHA` is empty and spec requires implementation changes, verdict MUST be REJECT (missing requirements), not a no-op DONE."
3. **Researcher:** Add post-Write step: "after Write tool call, confirm file exists via Read or Glob before claiming DONE."
4. **Debugger:** Add explicit instrumentation gate: "before returning DONE, confirm at least one of: (a) console output captured from reproduction run, (b) failing test saved to disk, (c) code path instrumented with probe logs. All-reading-no-instrument diagnosis must be explicitly tagged DONE_WITH_CONCERNS, not DONE."

## Q3 — What is the false-positive rate of `git diff --name-only` vs declared scope as a Builder sanity check?

### Source comparison table

| Source | Type | URL | Verified |
|---|---|---|---|
| PF v2 F-V10 record | PF-internal | docs/PROJECT-PLAN.md line 58 | 2026-05-01 |
| PF v2 F-V7 record | PF-internal | docs/PROJECT-PLAN.md line 55 | 2026-05-01 |
| DEV.to — "AI coding agents lie" | Primary | https://dev.to/moonrunnerkc/ai-coding-agents-lie-about-their-work-outcome-based-verification-catches-it-12b4 | 2026-05-01 |
| DEV.to — "What agents still miss" | Primary | https://dev.to/moonrunnerkc/ai-coding-agents-can-verify-some-of-their-work-now-heres-what-they-still-miss-58mc | 2026-05-01 |
| Anthropic — Building Effective Agents | Primary | https://www.anthropic.com/research/building-effective-agents | 2026-05-01 |

### Findings

The proposed check is: "if `git diff --name-only` in declared scope returns zero files and status = DONE, downgrade to DONE_WITH_CONCERNS." This is the right signal but needs a legitimate-empty-diff whitelist to avoid false positives.

**Legitimate empty-diff DONE cases (should NOT trigger downgrade):**

1. **Analysis or research verdict:** Builder is dispatched to audit/analyze, with an explicit outcome of "no change needed" or "already correct." F-V10's root cause was specifically a silent implementation failure, not an analysis task. The dispatch verb matters: "analyze X" → empty diff expected; "implement X" → empty diff is suspicious.

2. **Docs-only or plan-only tasks:** Builder writes to `docs/` or `docs/plans/` only — the VCS diff has file changes, but `git diff --name-only HEAD~1 src/` is empty because no source was touched. The scope must be compared to the right tree, not the entire repo.

3. **Refactor that produces net-zero diff:** Unlikely for a meaningful refactor, but possible for a pure rename revert. Scope + intent must both be checked.

4. **Idempotent migration already applied:** Database migration Builder dispatched after a migration was already run — schema is correct, migration file was already created by a prior dispatch. Empty diff is correct here.

**False-positive rate estimate (qualitative, not quantitative):**
- For "implement" / "build" / "fix" dispatch verbs: empty diff is highly suspicious. In PF v2's empirical record, 2/2 empty-diff events (F-V7, F-V10) were genuine failures, not legitimate analysis verdicts.
- For "analyze" / "verify" / "audit" dispatch verbs: empty diff is expected and DONE is correct.
- For "docs only" tasks: scope filter needed — `git diff --name-only HEAD~1 -- <scope_files>` not `git diff HEAD~1`.

**External evidence on outcome vs. claim:**

> "The alternative is checking what actually happened instead of what the agent said happened."
> — DEV.to, moonrunnerkc (verified 2026-05-01)

> "proof of execution is not proof of correctness / proof of correctness is not proof of safe merge / and self report is not proof of anything"
> — DEV.to, CrisisCore-Systems commenter (verified 2026-05-01)

### Q3 Recommendation

The empty-diff gate should be **dispatch-verb-conditional**, not universal:

```
IF dispatch_verb IN ["implement", "build", "fix", "refactor", "migrate"]
  AND git diff --name-only HEAD~1 -- <declared_scope_files> == empty
  AND status == DONE
THEN force DONE_WITH_CONCERNS
     append: "0 files changed in declared scope. Verify dispatch completed implementation, not analysis."
ELSE (analysis/docs/audit verbs)
  empty diff is expected; do not downgrade
```

The scope filter (`-- <declared_scope_files>`) is important: repo-wide `git diff` may show changes from prior commits in a worktree; the check should be bounded to what the Builder was asked to produce.

## Detection-Layer Design Synthesis

Across the four questions, five distinct detection mechanisms emerge:

| Mechanism | Catches | Trigger | Applies to |
|---|---|---|---|
| **D1 — Dispatch-verb-conditional empty-diff gate** | Builder silent-DONE (F-V10, F-V7) | `git diff --name-only -- <scope>` == 0 AND verb in {implement, fix, build} | Builder finalization |
| **D2 — Real-user smoke step for P2/BC-3 class** | OS-level event-interleave races (F-V11) | Any feature touching a state-setter with closure-read; any drag/IME/pointer-capture feature | QA Stage 2 + `browser-driven-verification` skill amendment |
| **D3 — Post-Write file-existence confirmation** | Researcher silent artifact-missing | After every Write call, confirm file exists via Read/Glob | Researcher checklist |
| **D4 — Instrumentation-gate for Debugger DONE** | Debugger over-confident read-only diagnosis | Before DONE: confirm at least one of: console output, failing test, or probe log | Debugger checklist |
| **D5 — QA empty-diff auto-REJECT** | QA passing an empty-diff as no-violation | `git diff $BASE_SHA..$HEAD_SHA` empty AND spec requires changes → verdict REJECT | QA Stage 1 |

These five mechanisms form the v2.2 detection layer. They are all **post-execution checks** (not prevention), which is what PF v2 currently lacks. The prevention layer (gates, hard-gates, tier-selection) is strong; the detection layer (what catches it when prevention silently fails) is the gap.

### Overall Recommendation

The detection layer should be added as **five targeted amendments** to existing agent prompts and skills — not a new agent. The evaluator-optimizer pattern (Anthropic) supports a second-pass verification call, but adding a sixth agent ("DetectionAgent") would increase dispatch cost and introduce its own silent-DONE modes. The cheaper path is strengthening post-execution checks within the existing five agents using the mechanisms above. The highest-priority amendments in order of empirical severity:

1. **Builder empty-diff gate** (D1) — directly fixes F-V10. Verb-conditional to avoid false positives on analysis tasks.
2. **Real-user smoke for BC-3/P2** (D2) — directly fixes F-V11. `browser-driven-verification` SKILL.md amendment + QA Stage 2 rule.
3. **QA empty-diff REJECT rule** (D5) — closes QA's silent pass-through of F-V10-class failures.
4. **Researcher post-Write confirmation** (D3) — low-cost, closes artifact-missing silent-DONE.
5. **Debugger instrumentation gate** (D4) — closes confident-but-uninstrumented silent-DONE.
